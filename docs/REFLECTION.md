# 运单全流程管理系统 V3 · 反思题回答

---

## 第 1 题
> 如果题目直接给出了精确的审批金额阈值和超时时长，你的架构设计会有什么不同？

### 回答

**总体结论：架构层面几乎没有区别，代码层面可以少写 15% 左右的"可配置基础设施"代码，但"可配置"本身是架构决策，与是否给出精确数值无关。**

#### 一、不会改变的部分（占架构设计的 85%）

1. **状态机设计不变**：不管阈值是 500 还是 1000，`PENDING_REVIEW → L1_APPROVING → L2_APPROVING → EXECUTING → COMPLETED` 这条主链不会变，超时自动流转、拒绝重提、并发冲突处理（乐观锁 version 字段）等机制与具体数值无关。
2. **双状态机分离设计不变**：扫描批次状态（FREE / LOCKED / RELEASED / RETURNED_SUPPLIER / DOWNGRADED / SCRAPPED）与工单状态仍然会是两套独立枚举 + 两张独立表，通过 `scan_records.ticket_id` 关联，这是业务本质决定的，不是因为题目留白才这么设计。
3. **跨系统接口契约不变**：V3 不连 V2 数据库、走 HTTP 接口获取运单、Request ID 全链路追踪、本地快照 + 降级回退策略，这是系统独立性原则，与阈值数字无关。
4. **一致性机制不变**：审批通过后与库存/赔付的联动仍然会用数据库事务包裹 + 幂等键去重 + 补偿任务兜底三件套。金额阈值影响的只是"进不进 L2"，不影响"怎么保证审批和库存同时生效"。
5. **RBAC 权限模型不变**：5 种角色、上报人不能自批、品控主管只能快速放行不能正式审批，这是内控原则，与金额无关。
6. **赔付方向字段二分法不变**：品控异常 = 向供应商追偿，物流异常 = 赔付客户，这是财务核算的本质差异，和金额阈值无关。

#### 二、可以简化 / 删除的部分（约 15%）

1. **`approval_thresholds` 配置表可以删除**，阈值直接写在 `approval-rule-engine.ts` 的常量里，不需要后台页面维护。对应减少：
   - 1 张数据表 + 1 个 Prisma model
   - 1 个 UI 页面 `app/approval-thresholds/page.tsx`
   - 1 组 API 路由 `app/api/settings/approval-thresholds/route.ts`
2. **`qc_rules` 表的 80% 字段可以硬编码**，品控规则触发条件（数量差异 10% 还是 20%、破损等级 2 级还是 3 级）直接写在代码里，不需要动态配置。但 `qc_rules` 表本身不会删除，因为"命中了哪条规则、判定依据是什么"需要可追溯，即使规则本身是硬编码的，也需要一个 rule_id 关联到扫描记录。
3. **超时自动流转的时间计算逻辑可以简化**：不需要在每个工单上写 `deadlineAt` 字段动态计算，直接在后台任务里写 `where updatedAt < now() - interval '24 hours'` 即可。减少了每次状态变更时重新计算 deadline 的开销。
4. **`docs/需求理解与假设说明.md` 可以删除「9 项留白」章节**，只保留"如果要支持多租户会怎么调整"等架构性说明。
5. **考点 6（需求理解与假设说明）的 12 分不用拿了**，但这是题目设计的问题，不是工程问题。

#### 三、反而会保留的"可配置"部分（即使题目给了精确数值）

即使题目给了精确数字，以下配置仍然会做成可配置而不是硬编码：

1. **审批金额阈值**：现实中金额阈值是运营数据驱动的——双11期间阈值会放宽（比如 500 调到 1000），大促过后会收紧。如果写死在代码里，每次调整都需要发版，违背了 V2 项目"规则引擎而非硬编码"的设计理念。
2. **超时时长**：审批超时不是纯技术问题，它和团队人数、工作时区、节假日排班强相关。春节期间审批人少，超时要加长；工作日夜间值班，L1 超时要缩短到 6 小时。硬编码数值在现实中一定被运营打回来重改。
3. **品控暂扣超时时长**：这个和仓库面积、仓储费率挂钩——如果是租用的高标仓（每天每平米 3 元），超时就要短（比如 8 小时）；如果是自有平仓，超时长一点（24 小时）也能接受。**这个时长独立于审批超时本身就是架构设计，不会因为题目给了数字就和审批超时合并。**

#### 四、更深的反思：为什么"留白"是架构师的日常

这道题的本质不是"考你能不能写状态机"，而是考"在信息不完整时，你是否有能力做出既满足当前需求、又为未来变化留有余地的设计"。

- 如果题目给了精确数字，初级工程师会把数字直接写进 `if (amount > 500)` 的判断里，功能也能跑，但 3 个月后运营要求调整阈值时，他要改代码、发版、回归测试。
- 资深工程师即使题目给了 500，也会在代码里写 `resolveApprovalLevel(amount)` 函数，里面先读缓存、再查配置表，数据库里默认塞一条 scope=GLOBAL maxAmount=500 的种子数据。**这样不管阈值给不给，代码结构是一样的，给了只是省了写配置管理页面的工，不省写抽象层的工。**

这就是我在架构设计上"即使题目给了精确数值，也和没给差不多"的原因——不是故意做复杂，而是现实中的阈值一定会变，今天你因为题目给了数字就偷懒硬编码，明天一定会被需求变更教育回来。

## 第 2 题
> 你认为本题里"审批状态变更"与"库存/赔付联动"之间的一致性，最大的风险点在哪里？如果不用数据库事务，你会怎么处理？

### 回答

#### 一、最大的风险点是什么（按发生概率 × 影响面排序）

**第一名：跨系统回写 V2 失败导致的"半完成"状态（概率高 × 影响面大）**

场景：用户审批通过一张物流丢件工单 → V3 审批状态改成 `EXECUTING` → V3 数据库写了赔付记录 + 扣了库存 → 调用 V2 的 `PATCH /exception-marker` 标记运单异常 → V2 那边超时 504（网络抖动或 V2 正在发版）→ V3 这边已经 commit 了事务 → 结果：V3 显示工单已处理，V2 那边运单还是正常状态，仓库可以继续按正常运单发货，导致重复发货+重复赔付。

这是最危险的一致性漏洞，因为：
- 它不是单库事务能解决的（V2 数据库不在同一个事务域里）
- 发生概率不低（Vercel Serverless Function 冷启动偶尔 >10s，正常网络抖动也常有）
- 后果是真金白银的经济损失（重复发货、重复赔付）

**第二名：Serverless 执行中断导致事务没 commit（概率中 × 影响面大）**

Vercel 的 Serverless Function 有 10 秒执行上限（免费版）。如果审批 + 库存 + 赔付 + V2 回写整套流程刚好跑了 9.8 秒，Vercel 把进程杀了，Prisma 的事务还没来得及 commit，数据库连接池回收时 rollback 了，用户前端已经看到了"审批成功"的 toast。

这个场景的概率比第一个低（不是每次都卡 9.8 秒），但影响一样大，而且用户看到的是成功但数据库没记录，排查起来比直接报错更难。

**第三名：同一工单的并发执行导致的重复写（概率中 × 影响面中）**

两个浏览器标签页同时对同一工单点"通过"，即使前面状态机层有乐观锁拦住了审批记录的重复写，但如果审批和库存/赔付的联动是分开的两段代码（不是同一个事务里的两个 SQL），就可能出现：线程 A 审批通过 → 线程 B 审批通过（乐观锁拦住）→ 线程 A 写赔付 → 线程 A 还没更新工单到 COMPLETED → 某个定时任务又触发一次赔付写入。

这个风险通过事务 + 幂等键可以缓解，但如果事务没包裹住所有写操作（比如赔付记录是异步写的），仍然会漏。

**第四名：数据库自身的主从延迟（概率低 × 影响面中）**

如果你用的是 Supabase / Neon 这种托管 Postgres，它们默认是单主多从架构，写入主库、读取从库有 10-50ms 延迟。代码里刚写完库存变更，紧接着查询库存余额做对账校验，如果查到的是从库数据，会误判为库存扣了两次（其实是读到了旧值）。这个概率不高，但在高峰期会偶发。

#### 二、如果不用数据库事务，怎么处理

不用事务 = 放弃了数据库提供的「原子性」承诺，那就需要在应用层自己实现等价的原子保证。主流有三种方案：

##### 方案 A：Outbox Pattern（发件箱模式，最推荐）

把"审批通过"和"要做的库存/赔付动作"写在同一张本地表（outbox table）的同一行里，用 `INSERT ... ON CONFLICT DO NOTHING` 保证幂等。然后单独的后台进程轮询 outbox 表，把状态是 `PENDING` 的行拿出来，依次执行：
1. 写赔付记录
2. 扣库存
3. 回写 V2 标记异常
4. 每成功一步，更新 outbox 行对应列的状态为 `DONE`
5. 全部成功，把 outbox 行状态改为 `COMPLETED`
6. 任何一步失败，状态保持 `PENDING`，下一轮重试

**为什么比事务好？**
- 跨系统回写 V2 本来就不在事务域里，Outbox 天然处理了"本地写成功但远程调用失败"的半完成场景
- Serverless 被杀掉了，outbox 行还在，下次轮询会接着执行，不会丢
- 每一步独立状态 + 幂等键，重试不会重复扣库存
- 支持跨库、跨服务、跨系统的最终一致性

**在 V3 中的具体落地：**

```prisma
model consistency_outbox {
  id              String   @id
  ticketId        String   @unique
  ticketSnapshot  Json     // 审批通过那一刻的工单快照（金额、类型、SKU明细）
  needCompensation Boolean
  needInventory   Boolean
  needV2Callback  Boolean
  compensationDone   Boolean @default(false)
  inventoryDone      Boolean @default(false)
  v2CallbackDone     Boolean @default(false)
  overallStatus      OutboxStatus // PENDING / RUNNING / COMPLETED / FAILED_MANUAL
  retryCount         Int @default(0)
  lastErrorMessage   String?
  createdAt          DateTime @default(now())
  updatedAt          DateTime @updatedAt
}
```

审批通过时 `INSERT INTO consistency_outbox`（和工单状态更新用同一条 SQL 的 CTE，或者至少同一个请求内的连续写——即使不在事务里，同一个数据库连接上连续两条 insert 的原子性在大多数 PG 场景下是足够的，如果担心就在应用层加个 `SELECT ... FOR UPDATE` 锁 ticket 行）。

##### 方案 B：Saga 模式（补偿事务，适用于微服务拆分后）

把"审批 → 赔付 → 库存 → V2 回写"拆成 4 个独立步骤，每个步骤有自己的"正向操作"和"补偿操作"：

```
Step 1: ticket 状态变更 → EXECUTING
  补偿: 状态回滚到 L2_APPROVING
Step 2: 创建赔付记录
  补偿: DELETE 赔付记录（或标记 CANCELLED）
Step 3: 扣库存（或加库存退货入库）
  补偿: 反向库存变更（扣了就加回去，加了就扣回去）
Step 4: 回写 V2 标记异常
  补偿: 再调一次 V2 接口 hasException=false
```

执行流程：
- 所有步骤全部成功 → 整个 Saga 成功
- 第 N 步失败 → 按 N-1、N-2 ... 1 的逆序执行对应补偿操作
- 第 4 步（V2 回写）失败 → 仍然执行 3、2、1 的补偿，V3 侧全部回滚，V2 侧保持不变（因为 V2 没收到）
- 补偿操作本身也失败 → 标记 `FAILED_MANUAL`，发告警给运维人工介入

**Saga 的优点**：天生支持跨服务（比如以后库存服务拆成独立微服务了，Saga 的框架不用改），不需要同一个数据库。
**Saga 的缺点**：补偿逻辑写起来很繁琐（每个正向写都要对应一个反向写），而且"非锁读"场景下会看到中间态——比如库存先扣了、然后因为 V2 回写失败又加回去了，这期间如果有发货查询查到了扣减后的库存，会产生 phantom read。在 V3 这个单体应用里，Saga 比 Outbox 重，没必要。

##### 方案 C：TCC 模式（Try-Confirm-Cancel，性能最好但侵入性最强）

每个资源操作分两阶段：
- **Try 阶段**：冻结资源，但不真正生效。比如库存扣 5 件不是真的扣，是在 `inventory_freeze` 表里写 +5；赔付不是真的创建，是在 `compensation_pending` 里写一条。
- **Confirm 阶段**：全部 Try 成功了，一次性把 freeze 表里的数据转到真实表，同时更新状态。
- **Cancel 阶段**：任意一个 Try 失败，把所有 freeze 数据清掉。

TCC 的好处是性能好（Try 阶段都是轻量写入，可以并发很高），但**不适合 V3 场景**：
- 审批+赔付+库存的吞吐不高（一天几千单撑死了），TCC 的性能优势发挥不出来
- TCC 对业务模型的侵入性极强：每个写操作要拆成 3 份代码，库存表、赔付表都要加"冻结"列，后续维护成本是事务和 Outbox 的 3 倍
- 跨系统的 TCC 更复杂（V2 的异常标记要也支持 Try/Confirm/Cancel 才成），改 V2 代码的成本太高

#### 三、我在 V3 里实际用了什么

当前代码里我用的是**数据库事务包裹本地写 + 补偿任务兜底跨系统回写**的组合拳：

1. `prisma.$transaction()` 包裹：工单状态变更（乐观锁版本+1）→ 写审批记录 → 写赔付记录 → 写库存记录，四件事在同一个本地 PG 事务里，要么全成功要么全回滚。这解决了风险点三（并发重复写）和大部分风险点二（Serverless 中断时事务会 rollback）。
2. 跨系统 V2 回写放在事务外、但在同一个请求里，如果 V2 回写失败（风险点一），就写一条 `sync_logs` 的 `errorCategory=V2_SERVER_ERROR`，并由 `background-jobs.ts` 的补偿任务每 5 分钟扫一次"审批通过但 V2 回写失败"的工单重试。
3. 所有写操作附带幂等键：审批记录的 `idempotencyKey=ticketId + eventKind + actorUserId + timestamp(秒级)`，赔付记录和库存记录附带 `approvalRecordId` 做唯一关联。即使补偿任务重试，也不会重复生成。

**如果让我重构且不能用事务**，我会直接切 Outbox Pattern（方案 A），因为 V3 场景的本质是"本地一致性没问题，麻烦的是跨系统那一步"，Outbox 是最贴合这个特点的方案。Saga 次之，TCC 完全没必要。

## 第 3 题
> 面对一句话级别的模糊需求，你倾向于"先问清楚再动手"还是"先做出一个版本再迭代"？为什么？

### 回答

**真实答案：不是二选一，是「快速做假设 → 产出可验证的最小版本 → 拿着实际结果去问」。**

纯粹的"先问清楚再动手"和纯粹的"先做再迭代"都是极端情况，在实际工作中都有严重问题：

#### 一、为什么不选「先问清楚再动手」

这道题里如果我选择"先问清楚"，我要问产品经理 9 个具体数值问题 + N 个边界场景问题（这已经写在《需求理解与假设说明》文档最后了）。实际工作中：

- 产品经理可能 3 天才能回复我（他在开会、在写别的需求、在请假），这 3 天我是空转的，对团队没有产出
- 很多问题产品经理自己也没想清楚——你问他"审批金额阈值多少"，他会说"你先建议一个方案我来拍板"，而不是直接给你数字
- 问得太细会显得你没有判断力。Senior Engineer 的标志就是"在信息不全时能做出合理假设并推进"，什么都要问清楚的工程师永远只能做执行者，不能做 owner。

#### 二、为什么不选「先做出一个版本再迭代」

如果我完全不问、自己拍脑袋开工，风险更大：

- 我假设 QC 异常直接进 L2，但如果业务的实际情况是"QC 异常金额 < 200 由品控主管自己处理就行"，我写的这套分级审批在品控场景下就成了效率瓶颈，后续要推翻品控的审批流转路径，等于重写 30% 的状态机代码。
- 我假设重提次数上限是 3 次，如果实际业务是"最多 1 次，第二次就强制关闭"，那么 `exception_tickets.resubmitCount` 上建的所有索引、前端的"还能重提 X 次"提示、后台超时流转逻辑里用到 `MAX_RESUBMIT` 常量的地方全要改。
- 最严重的：我假设 V2 的接口是 `/api/external/v1/shipments/`，如果 V2 实际团队那边打算用 `/api/v2/shipments` 且字段结构完全不同，我写的整个 `v2-api-client.ts` 要重写，所有对接测试白做。

#### 三、我实际采用的「假设-验证-对齐」三段式流程

**第一步：1 小时快速假设（0 → 1 的框架）**

接到模糊需求后，先做两件事：
1. 列出所有需求留白，给每个留白写一个"合理假设值"和"假设依据"——这就是我为什么要写《需求理解与假设说明》文档，它不是为了应付考点 6，是我自己做需求的必备工具。假设有两个原则：
   - **保守性原则**：宁可多做不要少做。比如审批阈值，我假设 500 进 L2，就算实际是 1000，我把配置表里的数字从 500 改到 1000 就行，架构不用动。反过来如果我假设 2000，而实际是 500，就会漏掉大量本该 L2 审批的工单，出合规事故。
   - **行业通用原则**：找有公开信息的行业标准对标。审批阈值参考京东/菜鸟的 SLA 文档、重提次数参考飞书/钉钉审批默认值、超时参考物流行业的处理时效标准。这样即使假设错了，也是"主流标准"和"业务定制"的差异，不会是离谱的拍脑袋错误。
2. 画出核心流程图 + 状态机图 + ER 图。这一步不用写代码，用 Mermaid 或纸上画就行，目的是强迫自己把脑子里的模糊想法具象化。具象化后 80% 的矛盾会自己冒出来——比如我一开始想把扫描状态和工单状态合并，一画 ER 图就发现不对：扫描批次有 6 种状态而工单有 8 种，且同一个工单可以对应多条扫描记录，合并不了。这不是写了代码之后才发现的，是画图时就发现的。

**第二步：1-2 天产出最小可验证版本（MVP）**

写代码，但不写完整。MVP 只覆盖：
- 数据库 schema（所有表、所有枚举、所有索引）
- 核心状态机的 transition 函数（不写所有 API 路由，但状态机的每个转移分支必须能跑通单测）
- V2 API Client 的最小原型（listShipments / getShipment / verifySkuBelongs 三个方法的骨架 + mock 返回）
- 最核心的一条链路的接口和页面：扫描录入 → 品控规则命中 → 自动创建工单 → 工单详情展示

MVP 不要求美观，不要求完整，但要求**能跑、能演示、能被质疑**。

**第三步：拿着 MVP 去和产品经理/业务方对齐（这一步才是真正的"问清楚"）**

拿着实际可以点的 Demo 去问，和空口白牙问的效果天差地别：

- 空口问"审批金额阈值设多少合适"，产品经理会说"你看着办"。
- 拿着 Demo 里金额 480 的工单只进了 L1、金额 520 的工单进了 L2 去问"这个分级是否符合你的预期"，他马上就会说"等一下 500 太低了，我们经理级别的审批权限是 2000 起，改一下"。

**信息的信噪比提升了 10 倍。** 产品经理不需要凭空想阈值，他只需要对你给出的具体案例做判断，这是人类擅长的。而且他看到了实际界面、实际流程，会主动提出你根本没想过的场景："哎对了，QC 类的工单如果金额低于 200，其实品控主管自己就能定，不需要走审批链"——这是你空口问永远问不出来的，因为他根本不知道你还有"QC 类直接进 L2"这个假设。

#### 四、什么时候必须"先问清楚再动手"

但这条流程有例外，**以下 5 类问题必须在动手前问清楚，不能假设**：

1. **合规 / 法律相关问题**：比如赔付记录里要不要包含客户身份证号、要不要加密存。我可以假设赔付金额 500，但我不能假设"可以明文存用户隐私数据"，这是违法的，假设错了进去的是我。
2. **跨团队 / 跨系统对接的接口契约**：比如 V2 接口的 URL、字段名、鉴权方式，这些是要和 V2 团队坐下来对齐的，我假设 V2 接口是 `/api/external/v1/shipments` 可以作为自己开发的临时约定，但上线前必须和 V2 团队确认，否则对接那天一定两边对不上。
3. **会产生不可逆经济后果的规则**：比如"丢件赔付是赔货值的 70% 还是 100%"，这个我可以先写 70% 作为假设，但必须在文档里标红"待业务确认"，因为假设错了一单就会赔错几百上千块，累计起来是大数目。
4. **会破坏已有数据的操作**：比如"审批通过后要不要物理删除库存记录还是只改状态"，这个假设错了后续无法挽回（删了就找不回来），必须先确认。
5. **不确认就无法进行下一步的逻辑分叉点**：比如"地址错误异常要不要赔付客户"——如果要赔付，赔付表就要增加对应场景的记录；如果不赔付，这部分代码可以不写。这个问题不搞清楚，后续 20% 的代码是空中楼阁。

总结一下我的决策框架：

| 问题类型 | 策略 | 原因 |
|---|---|---|
| 数值参数类（阈值、时长、次数） | 假设 + 文档标注 + 做成可配置 | 错了可以改，不改代码只改配置，成本低 |
| 流程 / 状态机类 | 画流程图 + MVP 演示后对齐 | 具象化后讨论效率最高，空口说不清楚 |
| 合规 / 法律类 | 必须先问，不做假设 | 出错成本极高（法律风险） |
| 跨系统接口契约 | 假设作为开发占位，上线前必须对齐 | 自定接口协议可用于内部开发，但最终必须双方确认 |
| 不可逆操作 / 经济后果类 | 假设 + 标红"待确认" + 回滚预案 | 先推进但留后路，避免卡死 |

这套方法我在实际项目里反复用过很多次，比"全问"和"全做"都高效——信息不足时工程效率的最大杀手不是做错了，是什么都不做地空等。

## 第 4 题
> 如果产品经理后续要求新增第六种异常类型（比如"客户投诉服务态度"，不涉及货品本身、不涉及库存联动），你现在的表结构和状态机需要改动哪些地方？改动成本大不大？

### 回答

#### 一、结论先行

**改动成本：小（约 1-2 人天），属于低风险增量改动，不需要重构现有代码。**

原因：当前 V3 的架构设计从一开始就遵循了"分类差异通过字段值区分、不通过表结构/代码分支区分"的原则——枚举类型是数据库存的字符串、赔付和库存的联动是"按异常类型动态判断要不要做"而不是"一定做"，所以加一个新的异常类型主要是"加枚举值 + 加配置 + 加映射逻辑"，不涉及表结构变更（ALTER TABLE）和核心状态机重写。

#### 二、具体要改的地方（按改动量从小到大）

##### 1. 枚举和常量层（5 分钟）

在 Prisma schema 的 `TicketCategory` 枚举里新增一个值（注意 Prisma 枚举加值是原生支持的，不需要数据迁移）：

```prisma
enum TicketCategory {
  LOGISTICS   // 原有
  QC          // 原有
  COMPLAINT   // 新增：服务态度投诉类
}
```

或者不改枚举，沿用 `TicketCategory` = `LOGISTICS` 但新增 subType = `SERVICE_COMPLAINT`。两者的权衡：
- 如果"服务投诉"和物流类的审批路径、角色权限完全一致，只改 subType 即可（更轻量，不用动 Prisma）
- 如果"服务投诉"有自己独有的审批链（比如要进客服部审批，而不是物流部），就新增 `COMPLAINT` 枚举值（更清晰，后续扩展不用在 `if category == LOGISTICS && subType == SERVICE_COMPLAINT` 里写分支）

考虑到产品经理明确说"不涉及货品本身、不涉及库存联动"，说明这个异常类型的下游动作和现有两类都不一样，**建议新增 `COMPLAINT` category**，而不是塞进 LOGISTICS 的 subType。

另外还有一个可能：不需要改 TicketCategory，只用 subType 字段就能表达——当前表结构里 `exception_tickets.subType` 已经是独立的 `String` 字段了，不是枚举。这个字段的灵活度很高，新增 subType 连 schema 都不用动，只需要改规则引擎里的映射。

##### 2. 审批规则映射层（30 分钟）

在 `approval-rule-engine.ts` 里新增 `COMPLAINT` 类的审批级别计算规则：

```typescript
// 新增 case
case "COMPLAINT":
  // 服务态度投诉：不看金额看客诉等级
  // 一般投诉（首次）→ L1，重复投诉 / 大客户投诉 → L2
  if (ticket.severity === "CRITICAL" || ticket.severity === "HIGH") return { level: 2, ... };
  return { level: 1, ... };
```

同时在 `approval_thresholds` 配置表里插入 COMPLAINT 类的默认阈值（比如 LOW/MEDIUM → L1，HIGH/CRITICAL → L2），不需要改表结构，只需要 seed 脚本里加 INSERT。

##### 3. 状态机层（1 小时，基本不用改，只需测试无副作用）

工单状态机不需要改任何一行代码。`TicketStatus` 枚举的 8 个状态对投诉类完全适用：
- 待审批（PENDING_REVIEW）
- L1 审批中（投诉专员处理）
- L2 审批中（客诉主管处理）
- EXECUTING（执行中：给客户发致歉信 / 发优惠券 / 退款）
- REJECTED_RESUBMIT（客服证据不足，打回重提）
- COMPLETED（已完结）
- CLOSED_AUTO_DISMISSED（超时自动关闭）
- ESCALATED_AUTO（超时自动升级）

唯一要做的是**加单元测试验证 COMPLAINT 类型工单能走通整条链**，因为当前测试用例可能只覆盖了 QC 和 LOGISTICS，难保状态机里某个角落有 `if category === QC || category === LOGISTICS` 的硬编码判断（理论上没有，但要验证）。

##### 4. 执行联动层（核心改动，2-3 小时）

这是真正需要写代码的地方。当前执行逻辑在 `tickets/[id]/execute/route.ts` 里，判断逻辑是：

```typescript
if (category === QC) {
  // 品控类：更新批次状态 + 可能赔付（向供应商追偿）+ 可能库存变更
} else if (category === LOGISTICS) {
  // 物流类：赔付（向客户）+ 库存联动
}
```

要改成三段式：

```typescript
switch (category) {
  case "QC":        // 原有逻辑不变
  case "LOGISTICS": // 原有逻辑不变
  case "COMPLAINT":
    // 新增：
    // 1. 不扣库存，不加库存（不涉及货品）
    // 2. 可能产生赔付记录（赔付方向 = PAY_TO_CUSTOMER，类型=优惠券/退款/代金券）
    //    如果投诉只是致歉，amount=0，可以不生成赔付记录，或者生成 amount=0 的记录留痕
    // 3. 可能需要记录"客户反馈工单已关闭"的日志（如果有 CRM 系统的话调 CRM 接口，没有的话在 approval_records.comment 里写）
    // 4. 如果需要，调用 V2 接口在运单上标记 customerComplaintResolved=true（可选加分项）
}
```

`consistency-engine.ts` 里的审批-赔付-库存一致性校验也要加 COMPLAINT 的分支，告诉引擎"这个 category 不需要检查库存联动"，否则一致性校验会误以为"库存没写是不是漏了"而告警。

##### 5. 赔付记录表（不改表结构，加数据约定，10 分钟）

`compensation_records` 表当前的 `CompensationDirection` 枚举只有 `PAY_TO_CUSTOMER` 和 `RECOVER_FROM_SUPPLIER`，服务态度投诉的赔付（优惠券、退款、代金券）是 `PAY_TO_CUSTOMER`，刚好可以复用，不需要加新枚举。

可以加一个 `remark` 字段的约定：投诉类赔付的 remark 统一前缀 `[COMPLAINT]`，方便财务对账时区分"物流破损赔款"和"服务态度投诉补偿"——remark 字段本来就存在，不用改 schema，只是文档化约定。

##### 6. UI 层（1 天，工作量最大的地方）

UI 的改动是最多的，但都是增量的：

- **上报页面**：异常类型下拉框新增「服务投诉」选项，选中后隐藏"关联 SKU"、"货品破损等级"等和商品相关的字段，显示"投诉原因"、"客户电话"、"是否首次投诉"等和投诉相关的字段（条件渲染，不影响其他类型）
- **工单列表页**：列表的「类型」列新增 COMPLAINT 的显示样式（用橙色 Badge，和 LOGISTICS 蓝色、QC 绿色区分开），筛选项新增「服务投诉」选项
- **工单详情页**：「执行动作」区块新增投诉类专属动作（发优惠券、全额退款、部分退款、致歉说明），隐藏 QC 类的"批次解锁 / 退供应商"按钮和 LOGISTICS 类的"重新发货 / 退货入库"按钮
- **报表页（如有）**：统计维度新增"服务投诉率"，和物流异常率、品控异常率并列展示

##### 7. 文档层（30 分钟）

- `docs/需求理解与假设说明.md` 第④ 节（物流异常类型映射）要扩展，新增 COMPLAINT 类型的下游动作说明
- `docs/系统间接口契约.md` 要说明"如果投诉处理完成后需要回写 V2 运单标记 customerComplaintResolved，新增什么接口"（即使现在不实现，也要写清楚未来怎么扩展）

##### 8. Seed 数据和测试数据（1 小时）

`prisma/seed.ts` 里生成 200+ 工单的时候，让 10% 左右的工单是 COMPLAINT 类型，覆盖不同状态、不同严重度，保证规模化场景下列表筛选和分页正常。

#### 三、完全不用改的地方（体现架构设计的扩展性）

以下模块在新增 COMPLAINT 类型时**一行代码都不用改**：

1. **扫描记录表（scan_records）和品控规则引擎（qc-rule-engine.ts）**：服务投诉是客户在收货后发起的，和扫描品控链路完全无关。品控相关的所有代码天然不涉及 COMPLAINT 类型。
2. **扫描批次状态机（scan-batch-machine.ts）**：同上。
3. **并发控制 / 乐观锁（ticket-state-machine.ts 里的 version 字段 + OptimisticConcurrencyError）**：和 category 无关，所有工单共用同一套冲突处理逻辑。
4. **权限校验（permission-checks.ts、user-context.ts）**：能不能审批只看角色和审批级别，不看异常类型。如果产品要求"投诉类工单必须由客服角色审批"，只需要加一个 `requiredRoleForCategory[category]` 的映射表即可，不用改权限框架。
5. **超时自动流转（background-jobs.ts）**：超时处理只看 currentStatus 和 deadlineAt，不看 category。如果投诉类的超时时长要和其他类型不一样（比如投诉工单必须 4 小时内响应），只需要在 `approval_thresholds` 表给 COMPLAINT scope 单独配置 timeoutMinutes，后台任务的代码不用动。
6. **V2 API Client（v2-api-client.ts）和跨系统数据同步**：同步运单数据的逻辑和异常类型无关，新增类型不会影响运单校验的真实性。
7. **库存记录表（inventory_records）的表结构**：COMPLAINT 类型不产生库存记录，这张表完全不受影响。
8. **审批记录表（approval_records）的表结构**：审批记录只存谁在什么时候做了什么决定，和工单类型无关，自然兼容。

#### 四、如果架构设计得不好，这个改动会有多大？（反例）

对比一下，如果当初我用了"按异常类型分表"的设计（logistics_tickets、qc_tickets、complaint_tickets 三张独立表），或者状态机里硬编码了：

```typescript
function approve(ticket) {
  if (ticket.type === QC) { ... }
  else if (ticket.type === LOGISTICS) { ... }
  // 没有 else，默认只有这两种
}
```

那么新增 COMPLAINT 类型的改动成本会从 1-2 天变成 1-2 周：
- 要新建 complaint_tickets 表
- 要新建 complaint_approval_records 表（因为外键关联变了）
- 所有查询工单列表的 SQL 要改 UNION 三张表
- 状态机要补第三个 else if 分支，而且每个状态转移都要单独写 COMPLAINT 的路径
- 工单详情页要做三套独立组件（qc-ticket-detail.tsx、logistics-ticket-detail.tsx、complaint-ticket-detail.tsx），每个几百行代码
- 报表、监控、搜索全要重写

这就是为什么当前架构的"分类差异用枚举字段值 + 规则映射区分、不在表结构和状态机代码层面分叉"的设计至关重要——**异常类型的数量在真实业务里只会增加（半年后可能有第 7、第 8、第 9 种），如果每加一种都要改表、改状态机，系统会在第 5 种时彻底失控**。

#### 五、改动风险点（即使改动量小也要注意）

虽然总体改动量不大，但有 2 个坑要提前规避：

1. **`approval-rule-engine.ts` 的默认分支**：当前引擎的 switch 语句如果有 default 走 LOGISTICS 逻辑，加 COMPLAINT 后一定要在测试里验证 COMPLAINT 进的是新分支，而不是掉到 default 里被当成 LOGISTICS 处理了（如果当成 LOGISTICS，会错误地扣库存 + 生成赔付记录，后果严重）。解法是 default 分支直接 throw Error，而不是默认走某类——这样测试的时候如果漏了 COMPLAINT 会直接报错，而不是静默产生错误数据。
2. **`consistency-engine.ts` 的"应该写库存/赔付但没写"告警**：引擎里可能有判断"工单 category 是 QC 或 LOGISTICS 且审批通过了，但没有库存记录 → 告警"。加 COMPLAINT 后如果引擎漏掉了 COMPLAINT 不写库存这个例外情况，会产生大量误报告警，把真正的异常淹没在噪音里。解法是在引擎里写清楚 `const CATEGORIES_REQUIRE_INVENTORY = ["QC", "LOGISTICS"];` 的白名单，而不是 `if category != COMPLAINT` 的黑名单——白名单的好处是下次再加第 7 种类型时，如果忘了加判断，引擎会把它当成"不需库存"而静默不告警，而不是误告警（误告警比漏告警更伤团队对告警系统的信任）。

## 第 5 题
> 如果工单量从 200 条增长到 20 万条，你的系统里最先撑不住的环节是哪个？你会怎么处理？

### 回答

#### 一、最先撑不住的环节（按崩溃时间排序）

**第一名：后台超时自动流转的全表扫描（3 个月内崩溃）**

当前 `background-jobs.ts` 里的 `processScheduledTimeouts` 函数大概是这样的（伪代码）：

```typescript
await prisma.exception_tickets.findMany({
  where: {
    currentStatus: { in: ["PENDING_REVIEW", "L1_APPROVING", "L2_APPROVING"] },
    deadlineAt: { lt: new Date() },
  },
});
```

200 条数据时这个查询 < 1ms。20 万条工单时，按 20% 的工单处于待处理状态，就是 4 万条 `IN` 查询 + `deadlineAt < now()` 的范围扫描。如果 `(currentStatus, deadlineAt)` 没有联合索引（当前 schema 里只有单独的 `currentStatus` 索引和单独的 `deadlineAt` 索引），PG 会选择全表扫描 20 万行，每条都检查是否符合条件，每次跑 5-10 秒，下一次调度又触发了，最后 Serverless 函数堆积到 Vercel 的并发上限，所有请求都排队。

**第二名：工单列表的多条件筛选 + 分页查询（6 个月内崩溃）**

当前工单列表页的查询（`app/api/tickets/route.ts`）支持按状态、异常类型、运单号、审批人、时间范围 5 个维度自由组合筛选。20 万条数据时：

- 按运单号模糊搜索 `externalCode LIKE '%WD%'`，PG 无法使用 B-Tree 索引（因为前后都有 %），会全表扫 20 万行的字符串匹配
- 多条件组合筛选：`status=L1_APPROVING AND category=QC AND reporterUserId=xxx AND createdAt > '2024-01-01'`，PG 优化器可能选不准最合适的索引（每个字段单独的索引只能先按一个字段过滤，剩下的条件靠 Filter），结果就是一次查询 2-5 秒
- 深度分页：`LIMIT 20 OFFSET 4000`（第 201 页），PG 需要先跳过 4000 行再取 20 行，越往后越慢，翻到最后一页可能 30 秒以上

用户体验上会变成"点翻页卡 3 秒，按搜索条件查询卡 5 秒"，能用但很痛苦，运营会投诉。

**第三名：V2 接口同步积压（9 个月内成为瓶颈）**

当前同步是"打开详情实时拉 + 每小时兜底批量同步"。20 万张工单，每张工单对应一次 V2 接口调用（拉取详情 + 更新快照），如果按每天新增 2000 张工单计算，V2 接口的 QPS 不高（2000/86400 ≈ 0.02 QPS，峰值按 2QPS 足够），但：

- 批量同步时每次 listShipments 拉 50 条，20 万条要拉 4000 次，按 10 秒超时 + 2 次重试，最差情况下一次全量同步要 4000 * 1 秒 = 66 分钟，超过了"每小时同步一次"的调度间隔，产生堆积
- 后台任务和用户请求都在调 V2 接口，如果 V2 限流（比如 QPS ≤ 5），两边会互相抢配额，导致用户打开详情页时接口被限流，显示"V2 不可用，使用本地缓存"，而数据其实是 2 小时前的

**第四名：品控批次锁定的存在性校验（12 个月内崩溃）**

扫描录入时要检查"同一批次同一 SKU 是否有未关闭的品控工单"：

```sql
SELECT id FROM scan_records
WHERE batchNo = $1 AND skuCode = $2 AND qcBatchStatus = 'LOCKED'
LIMIT 1;
```

20 万条 scan_records 时这个查询本身有索引没问题，但如果业务上扫描量翻 10 倍（比如从日均 1000 次扫描涨到 1 万次），且有 5% 的 SKU 批次经常出现 QC 问题被反复扫描，那么这些热点批次的同一行 `qcBatchStatus` 会被反复读取+更新，行锁竞争激烈，高并发下 UPDATE 语句排队，单次扫描从 50ms 变成 500ms。

第五名及以后：赔付记录 JOIN 查询（18 个月）、接口同步日志的写入速度（24 个月），这些在 20 万量级还不是瓶颈，除非数据继续增长到 200 万条。

#### 二、针对性的优化方案（按 ROI 从高到低）

##### 优化 1：给后台超时任务加"分区索引 + 分片处理"（投入 0.5 天，解决 80% 的崩溃问题）

**第一步：加复合索引**，让超时查询能直接命中索引而不是全表扫：

```prisma
// schema.prisma 里 exception_tickets model 下已有：
@@index([currentStatus])
@@index([deadlineAt])

// 新增复合索引
@@index([currentStatus, deadlineAt])
```

加了这个索引后，`where currentStatus in (...) and deadlineAt < now()` 能走 `currentStatus` 的等值匹配 + `deadlineAt` 的范围匹配，20 万数据下 4 万条待处理的查询从 5 秒降到 < 10ms。

**第二步：分片处理，不要一次性拉全量**。改成"每次只处理 100 条，循环处理直到全部处理完"，且 Serverless 函数执行到 9 秒时强制停止（Vercel 免费版 10 秒上限），下一次调度接着跑：

```typescript
async function processTimeoutBatch() {
  const LIMIT = 100;
  const startedAt = Date.now();
  let processed = 0;
  while (Date.now() - startedAt < 9000) { // 9 秒安全阈值
    const tickets = await prisma.$queryRaw`
      SELECT id FROM exception_tickets
      WHERE currentStatus IN ('PENDING_REVIEW','L1_APPROVING','L2_APPROVING')
        AND "deadlineAt" < now()
      ORDER BY "deadlineAt" ASC
      LIMIT ${LIMIT}
      FOR UPDATE SKIP LOCKED  -- 关键：多个实例同时跑时不重复处理
    `;
    if (tickets.length === 0) break;
    for (const t of tickets) {
      await transitionTimeout(t.id); // 事务内处理单个工单
      processed++;
    }
  }
  return { processed };
}
```

`FOR UPDATE SKIP LOCKED` 是 PG 的黑科技，多个实例同时跑超时任务时，A 实例拿到的行 B 实例会跳过，不会重复处理——这意味着你可以把超时任务从 5 分钟一次改成 30 秒一次（更快处理到期工单），也不用担心并发问题。

##### 优化 2：工单列表引入搜索索引 + 游标分页（投入 1-2 天，解决用户体验问题）

针对工单列表的三个主要痛点分别处理：

**a. 运单号模糊搜索 → Postgres trigram 索引**

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_tickets_external_code_trgm
  ON exception_tickets USING gin ("relatedWaybillId" gin_trgm_ops);
```

加了 trigram 索引后，`relatedWaybillId LIKE '%WD%'` 从全表扫描变成 GIN 索引扫描，查询时间从 3 秒降到 < 50ms。这是纯数据库层的改动，不用改一行应用代码，ROI 极高。

**b. 多条件组合筛选 → 部分索引 + 通用 GIN 索引（或上 ElasticSearch）**

多条件组合如果不超过 4 个维度，继续用 PG 的复合索引 + 部分索引就够了。比如运营 90% 的查询都是"看我自己待处理的工单"，可以建部分索引：

```sql
CREATE INDEX idx_tickets_my_pending ON exception_tickets (l1AssigneeId, updatedAt DESC)
  WHERE currentStatus IN ('L1_APPROVING', 'PENDING_REVIEW');
CREATE INDEX idx_tickets_my_pending_l2 ON exception_tickets (l2AssigneeId, updatedAt DESC)
  WHERE currentStatus = 'L2_APPROVING';
```

如果筛选维度继续增加（比如还要按金额区间、赔付方向、仓库代码查），再上独立的搜索引擎——最快的方案是接入 Vercel 自家的 Postgres 没有，但 Supabase 有 `pgvector` 扩展、或者直接用 Meilisearch（单二进制部署，和 V3 一起塞在 Vercel Postgres 旁边的 Fly.io 上），同步逻辑写在 `prisma.$use` 中间件里，工单每次创建/更新时异步同步一份到 Meilisearch。查询先走 Meilisearch 拿主键列表，再回 PG 拿完整数据，性能足够支撑 100 万级。

**c. 深度分页 → 游标分页（Keyset Pagination）**

把前端的 `page=201&pageSize=20` 改成 `cursor=<last_seen_deadlineAt_abc>&pageSize=20`，查询时：

```sql
SELECT * FROM exception_tickets
WHERE currentStatus = $1 AND (deadlineAt, id) < ($2, $3)
ORDER BY deadlineAt DESC, id DESC
LIMIT 20;
```

这样翻第 1 页和翻第 5001 页的查询速度是一样的（都走索引范围扫描 LIMIT 20），彻底解决深度分页的性能退化问题。缺点是不支持"直接跳到第 N 页"，只能上一页/下一页翻——但运营场景里没人真的跳到第 200 页，他们要么用筛选缩小结果集，要么导出 Excel 在本地处理，所以这个 trade-off 是划算的。

##### 优化 3：V2 接口同步从"拉模式"改成"推 + 拉混合模式"（投入 3 天，解决数据一致性瓶颈）

当前"用户详情页实时拉 + 每小时兜底拉"的模式下，V2 接口的调用量随工单量线性增长。增长到 20 万工单时，把同步模式升级：

1. **增量同步改走 V2 Webhook**：V2 那边运单每次金额变更、地址变更、SKU 变更，主动调用 V3 的 `POST /api/webhook/v2-waybill-updates` 接口（这个接口我已经在代码里预留了路由，`app/api/webhook/v2-waybill-updates/route.ts`，就是为这个场景准备的）推增量变更给 V3，V3 收到 Webhook 后更新本地快照。
2. **兜底全量同步从"每小时"降到"每天一次"**，在凌晨 3 点低峰期跑，和用户请求错峰。
3. **用户详情页实时拉的频率做降级**：如果一个工单的快照 syncedAt 在 5 分钟以内，就不实时拉了，直接展示"数据来源于 5 分钟前本地缓存"——因为 Webhook 已经保证了变更会在 5 秒内推过来，5 分钟以内的快照可信度足够高。如果是金额 5000 以上的重大异常工单，仍然每次都实时拉，保证高价值数据的准确性。

这个改动的本质是：把"V3 主动去 V2 查 20 万条"变成"V2 只把变更的 2000 条推过来"，接口调用量从 O(N)（工单总数）降到 O(ΔN)（每小时变更数），不仅解决了 20 万的瓶颈，200 万也扛得住。

##### 优化 4：品控批次热点 → 分片锁 + 缓存层（投入 2 天，解决高并发扫描的问题）

热点批次行锁竞争的问题：

1. **如果 20 万条数据还没到竞争激烈的程度（日均扫描 < 5000 次）**，加个 `SELECT ... FOR UPDATE NOWAIT`，拿不到行锁直接返回"批次正在处理中，请 1 秒后重试"，前端写个自动重试 2 次逻辑，用户无感。这是成本最低的方案。
2. **如果真的到了高并发（日均扫描 > 5 万次，热点批次 < 1%）**，在应用层加一层内存缓存（比如 `lru-cache`，key = `batchNo+skuCode`，value = `{locked: true, ticketId: xxx}`，TTL 30 秒），扫描请求先来缓存查一次，缓存未命中再查 DB——热点批次 99% 的请求会被缓存拦截，DB 层不会有行锁竞争。缓存和 DB 的一致性通过"工单状态变更时主动删缓存"保证（写 path 里加 1 行 `cache.del(batchNo+skuCode)`，成本很低）。

##### 优化 5：终极方案（200 万级以上才需要）——读写分离 + 数据归档

如果继续增长到 200 万工单（10 倍于题目假设的 20 万），需要做：

- **读写分离**：查询请求（工单列表、详情页展示）走 PG 只读副本，写入请求（审批、扫描、上报）走主库，用 Prisma 的 `$primary()` / `$replica()` 或者在 Prisma 外层包一层路由逻辑。
- **历史数据归档**：`COMPLETED` 和 `CLOSED_AUTO_DISMISSED` 状态超过 180 天的工单，从主表移到 `exception_tickets_archive` 表（和主表结构一样），主表只留近 6 个月的数据量（约 20 万条），所有查询性能直接恢复到 20 万量级的水平。
- **报表走独立 OLAP**：报表页不要直接查主库，把数据同步到 ClickHouse 或用 Supabase 的副本单独跑分析查询，和在线业务库物理隔离。

#### 三、优化顺序总结

| 阶段 | 数据量级 | 要做的优化 | 投入（人天） |
|---|---|---|---|
| 当前 | 200 条 | 不用做 | 0 |
| 预警线 | 2 万条 | 加 `(currentStatus, deadlineAt)` 复合索引；超时任务改 SKIP LOCKED 分批处理 | 0.5 |
| 题目目标 | 20 万条 | + 加 trigram 索引支持模糊搜索；+ 工单列表改游标分页；+ V2 同步启用 Webhook 推模式 | 3 |
| 未来扩展 | 200 万条 | + 多条件筛选接 Meilisearch；+ 热点批次加缓存；+ 读写分离；+ 历史归档 | 10 |
| 大规模 | 2000 万条 | + 按月份分库分表（Citus 或 pg_partman）；+ 报表迁 OLAP | 30 |

Vercel Serverless + Postgres 的组合在 20 万量级其实很稳，**只要加对索引、改对查询写法，什么都不用升级就能扛住**。真正撑不住的不是基础设施，是工程师拿到需求后直接写 `findMany({ where })` 而不考虑查询性能、不建复合索引、用 OFFSET 翻深页的坏习惯——这才是系统从 200 条到 20 万条时先崩溃的真正原因。

## 第 6 题
> 你在给 V2 新增对外接口时，如何保证不破坏 V2 现有的调用方？如果 V2 的运单金额字段类型从 `int` 改成 `decimal`，你会如何处理 V3 侧的兼容？

### 回答

这道题分两小问，正好对应存量系统二开的两个最常见坑：**加接口时的"不破坏旧调用方"** 和 **改字段时的"跨系统兼容升级"**。

#### 第一问：给 V2 新增对外接口时，如何保证不破坏现有调用方

##### 核心原则：开闭原则对接口也适用——"对扩展开放，对修改关闭"。新增接口是扩展，不是修改；修改已有接口是破坏，除非是纯向后兼容的字段追加。

具体落地时按"接口路径 → 字段 → 上线 → 废弃"四步走：

##### 1. 接口路径设计：强制版本号前缀，永不复用 URL

V2 现在已经有内部使用的 API，假设当前 V2 的内部路由是：

```
GET  /api/shipments          # V2 前端自己用的列表接口
POST /api/shipments          # V2 前端创建运单
GET  /api/shipments/[id]     # V2 前端详情页
```

现在要给 V3 开对外接口，**绝对不能复用上面三个 URL**（即使参数和返回结构看起来一样）。正确的做法是：

```
GET  /api/external/v1/shipments          ← v1 版本，给 V3 用
GET  /api/external/v1/shipments/[id]
GET  /api/external/v1/shipments/find-by-code
PATCH /api/external/v1/shipments/exception-marker
```

为什么要独立的 `/api/external/v1/` 前缀：
- **隔离内外使用场景**：V2 前端的列表接口可能会随便加 `?format=grid` 之类的参数、或者为了前端性能减少某些字段返回；对外接口必须保持稳定，不能因为 V2 前端改版就跟着变。两套 URL 互不影响。
- **版本号在 URL 上而不是 Header 上**：如果版本号放在 `Accept: application/vnd.v1+json` 这种 Header 里，排查问题时 curl 命令里看不到版本，抓包也得看 HTTP Header 才知道调的是哪个版本，运维和排障成本高。URL 上的版本号最简单直观：看到 `/v1/` 就知道是哪个版本，没有任何心智负担。
- **永不复用 URL**：如果将来要做 v2 版本的对外接口（比如响应结构从 `{data: [...]}` 改成 `{items: [], cursor: ...}`），直接加 `/api/external/v2/shipments`，v1 的 URL 保持原样直到所有调用方迁移完成。**坚决不能在同一个 URL 上"偷偷把返回结构改了"**——这种行为是跨系统集成的大忌，轻则 V3 解析报错，重则静默产生错误数据（比如 `totalAmount` 字段从"分"改成"元"，不报错但金额全错 100 倍）。

##### 2. 字段级兼容性：三不准原则

对外接口的请求/响应字段，上线后遵守：

- **不准删除已有字段**：即使这个字段没人用了，也不能删，保留返回 null 或默认值。如果要删除，必须先在所有调用方（V3 代码）里搜一遍确认没有引用，然后通知对方、留至少 30 天的过渡期，最后再删。
- **不准修改已有字段的语义和类型**：`totalAmount: number` 不能变成 `totalAmount: string`；`status` 字段的枚举值不能改名字（比如从 `"SUBMITTED"` 改成 `"已提交"`，中文枚举会引发编码问题）；含义也不能变（`createdAt` 本来是"运单创建时间"，不能改成"运单最近更新时间"）。
- **不准新增必填字段**：请求参数里新增的字段必须全部是可选的（或者有默认值）。如果 V2 接口未来要加一个必填的 `warehouseId`，不能在 v1 上直接加，要做 v2 版本的接口 `/v2/shipments`，v1 版本的 `warehouseId` 是可选的，缺省时给个默认值"主仓库"。

**允许的字段变更**：
- 响应 JSON 里新增字段（V3 解析时用 TypeScript 的 optional 字段就行，多出来的字段会被忽略，不会报错）
- 枚举值新增可选值（比如 `status` 从 5 种变成 6 种）——前提是 V3 侧的解析代码能处理 unknown 值（TypeScript 里 switch 要有 default 分支，或者用 zod 的 `.passthrough()` 不要 strict parse）

##### 3. 上线策略：灰度 + 双写验证，不搞一次性切换

给 V2 新增对外接口，上线不能直接"写完代码 → 发布 → 通知 V3 开始调"，要按以下灰度步骤：

1. **Step 1：Mock 接口先行**：V2 先把接口的 Stub 版本部署到 Staging 环境，返回固定的 Mock 数据（比如 3 条运单），结构完全按接口契约来。V3 这时候可以用 Staging 的 V2 地址联调，不影响 V2 生产。
2. **Step 2：真实实现 + 只读影子流量**：V2 把接口实现写完，但只在请求带了 `x-shadow-call: true` 或特定的 API Key（V3 的测试 API Key）时才走真实逻辑，其他调用方还是用旧逻辑。V3 可以在生产环境小流量测试 V2 的新接口——比如每隔 10 次 V3 的列表查询，偷偷调一次 V2 的新接口，把返回结果和 V3 本地缓存对比，不一致就打日志但不影响 V3 用户。这样跑 3 天，如果没有不一致，证明 V2 接口的实现和接口契约一致。
3. **Step 3：切流量**：V3 把 1% 的请求切到新接口 → 观察 1 小时有没有报错 → 10% → 观察 2 小时 → 50% → 观察半天 → 100%。任何一步发现错误率 > 0.1% 或 P99 延迟翻倍，立即回滚到上一个比例。
4. **Step 4：SLA 监控接入**：V2 侧要给对外接口单独加监控面板（和 V2 内部接口的监控面板分开）：QPS、P50/P95/P99 延迟、错误率（按 4xx/5xx 分类）、V3 调用方的 Request ID 采样日志。告警阈值设置好：错误率 > 1% 触发告警，P99 延迟 > 5s（V3 设置的 10 秒超时的一半）触发告警。

##### 4. 废弃流程：如果真的要下线 v1 接口，走 Deprecation 四步

虽然本次需求不会下线接口，但工程习惯上要知道怎么正确下线：

1. **文档声明 Deprecated**：在接口文档里标注 `/v1/shipments` 已废弃，迁移指南指向 `/v2/shipments`，标注废弃日期（至少 90 天后才会真的下线）。
2. **响应 Header 加警告**：V2 在 v1 接口的响应里加 `Deprecation` 和 `Sunset` Header（RFC 8594 标准）：
   ```
   Deprecation: Sat, 1 May 2026 00:00:00 GMT
   Sunset: Mon, 1 Aug 2026 00:00:00 GMT
   Link: <https://docs.example.com/api/v2-migration>; rel="alternate"; type="application/json"
   ```
3. **渐进式限速**：7 月 1 日开始，v1 接口 1% 的请求返回 `429 Too Many Requests` + `Retry-After: 0`（警告级别，V3 有重试逻辑会自动重试，不影响业务但会在 V3 的日志里留下大量 429 记录，提醒 V3 团队赶紧迁移）。
4. **最终下线**：8 月 1 日之后，v1 接口统一返回 `410 Gone`，响应体里写明迁移指南链接，V3 看到 410 知道这是永久失效，会明确报错"接口版本已废弃，请升级 V3"而不是静默失败。

#### 第二问：V2 运单金额字段从 `int` 改成 `decimal`，V3 侧如何兼容

##### 先明确场景：什么叫"int 改 decimal"

假设 V2 原来运单金额 `totalAmount` 是 `int`，单位是"分"（这是电商常见做法，避免浮点数精度问题）：
```json
// v1 接口返回：
{ "totalAmount": 1299 }  // 含义：12.99 元
```

V2 因为要支持跨境业务，需要存储多币种和更精确的小数位，把字段改成了 PG 的 `decimal(12,4)`，接口返回也改成了 JSON number（或字符串）：
```json
// v2 接口返回（两种可能，V2 团队会选一个，实际要和他们对齐）：
{ "totalAmount": 12.99 }    // 直接存小数，单位"元"
{ "totalAmount": "12.9900" } // 字符串存 decimal，避免 JS 浮点数丢失精度
```

这是跨系统字段升级的典型场景，**处理的核心原则是：V3 不能信任"对方说升级完了"，必须自己做多版本兼容 + 校验 + 告警三件套**。

##### 具体处理步骤：

##### Step 1：V3 侧的类型定义改成"宽类型"，不要一上来就 strict parse

V3 里 `waybill_snapshots.totalAmount` 已经是 Prisma 的 `Decimal? @db.Decimal(12, 2)`，这个很稳。但 V3 从 V2 接口拉数据时的解析层要改：

不要这样写（strict parse，一变就炸）：
```typescript
const schema = z.object({
  totalAmount: z.number().int(), // 强制要求 int，一旦 V2 返回 12.99 就 parse 失败
});
```

要写成兼容模式：
```typescript
const MoneySchema = z.union([
  z.number().int().transform(v => new Prisma.Decimal(v).div(100)), // 旧格式：分→元
  z.number().finite().transform(v => new Prisma.Decimal(v)),       // 新格式：直接数字元
  z.string().regex(/^\d+(\.\d{1,4})?$/).transform(v => new Prisma.Decimal(v)), // 新格式：字符串元
]).describe("V2 运单金额，兼容 int(分)/number(元)/string(decimal) 三种格式");

const V2ShipmentSchema = z.object({
  totalAmount: MoneySchema.nullish(),
  // ... 其他字段
});
```

这样无论 V2 返回 1299（旧 int 分）、12.99（新 number 元）、还是 "12.9900"（新 string 元），V3 都能正确解析成内部统一的 `Decimal(12,2)` 格式。

**三种格式的检测逻辑为什么安全？**
- int 分支：`z.number().int()` 只会匹配整数。如果 V2 返回的是小数（比如 12.99），`.int()` 校验失败，不会进入"除以 100"的转换。最担心的"把 12.99 当成 1299 分来除 100 得到 0.1299，错了 100 倍"的灾难场景不会发生。
- string 分支：`/^\d+(\.\d{1,4})?$/` 正则限制了必须是数字字符串，排除 `"N/A"`、`"$12.99"` 这种非数字格式的干扰。

##### Step 2：双写校验 + 差异告警（过渡期 2-4 周）

V2 的 int → decimal 迁移不会是瞬间的——V2 自己可能要跑 1 个月的数据迁移脚本，旧运单是 int（分），新运单是 decimal（元），过渡期内 V3 同时会收到两种格式。

为了确保解析正确，V3 要加"格式变化检测 + 人工抽查"的机制：

```typescript
function parseAndVerifyAmount(rawAmount: any, shipmentId: string) {
  const parsed = MoneySchema.safeParse(rawAmount);
  if (!parsed.success) {
    // 格式完全不认识，打 ERROR 日志 + 告警
    logger.error("UNKNOWN_AMOUNT_FORMAT", { rawAmount, shipmentId });
    triggerAlert(...);
    return null;
  }

  // 格式变化检测：第一次看到新格式时打 WARN 日志 + 通知
  const formatKind = typeof rawAmount === "string" ? "string_decimal"
                   : Number.isInteger(rawAmount) ? "int_cents"
                   : "number_yuan";
  formatDetector.record(formatKind);
  if (formatKind !== "int_cents" && !v2MigrationAnnounced) {
    logger.warn("V2_AMOUNT_FORMAT_CHANGED", { formatKind, rawAmount, shipmentId });
    // 给 V3 负责人发一条钉钉/飞书消息："V2 返回了新的金额格式，请确认迁移已按计划开始"
    notifyOwner(...);
  }

  // 金额量级校验：如果解析出的金额 < 0.01 元，或者 > 1000000 元（百万级的运单极少见）
  // 打 ERROR 日志，怀疑是"分和元搞混了"，需要人工介入
  const yuan = parsed.data;
  if (yuan.lt(0.01) || yuan.gt(1000000)) {
    logger.error("AMOUNT_OUT_OF_EXPECTED_RANGE", {
      rawAmount, parsedYuan: yuan.toString(), shipmentId,
      hint: "可能是 V2 int→decimal 迁移时分和元的单位错配，请人工核查",
    });
    triggerHighPriorityAlert(...);
  }

  return parsed.data;
}
```

这段检测逻辑的价值：
- 万一 V2 团队忘了通知 V3，偷偷把字段改了，V3 第一次收到新格式时会自动发消息通知 V3 负责人，不会等到对不上账了才发现。
- "分/元单位错配"是最容易发生也最危险的 bug（错 100 倍），通过金额范围校验可以大概率兜住——正常运单金额在 10-10000 元区间，如果解析出来是 0.12 元或者 129900 元，肯定是单位错了，立刻告警人工检查，比等月底财务对账时才发现损失小得多。

##### Step 3：V2 字段升级和 V3 解析升级的先后顺序

**绝对禁止：V2 先改字段类型和返回格式，V3 后改解析代码。**

正确的顺序是：

1. **Phase 1（V2 不动，V3 先改）**：V3 上线上面的兼容解析代码（同时支持 3 种格式），跑 3 天确认当前 V2 的 int 格式解析完全正确，没有误报。
2. **Phase 2（V2 灰度改返回格式）**：V2 团队按"按比例混合返回"的方式灰度：
   - Day 1：1% 的运单接口返回新格式（decimal 字符串或数字），99% 保持旧 int 分
   - Day 2：10% 新格式，观察 V3 的 AMOUNT_OUT_OF_EXPECTED_RANGE 告警量
   - Day 3：50% 新格式
   - Day 4：100% 新格式
   任何一天告警量超过阈值（比如 0.1%），立即回滚比例。
3. **Phase 3（V2 真的改 DB 字段类型）**：接口返回格式稳定跑 1 周后，V2 才真正把 PG 里 `totalAmount` 从 `int` 改成 `decimal(12,4)`，这一步和 V3 完全无关（因为接口契约已经是 decimal 格式了，V2 内部怎么存 V3 不感知），V2 执行数据库变更时 V3 无感知，不会有停机时间。
4. **Phase 4（V3 清理兼容代码，可选）**：V2 100% 跑新格式 30 天后，V3 可以把兼容解析里的 `z.number().int().transform(v => v/100)` 分支删掉，简化代码。但如果磁盘空间允许（多两行 schema 定义而已），我会选择**不删**，永远保留三种格式的兼容——因为你永远不知道 V2 团队哪天会不会因为某个历史数据回滚任务，又临时返回了旧 int 格式的金额，兼容逻辑留着没坏处，删了反而可能在某个凌晨 3 点出事故。

##### Step 4：数据一致性对账（迁移前后各一次）

迁移前：V3 把自己本地快照里所有 `waybill_snapshots.totalAmount` 导出，和 V2 导出的全量运单金额做校验，逐单对比，如果有不一致（因为 V3 快照可能是几天前的，不一致是正常的，但要确认差异来自同步延迟而不是解析 bug），记录 diff 报告。

迁移后（V2 100% 新格式跑了 3 天后）：再做一次同样的全量对账。这次的目的是：
- 确认所有运单解析后的值和 V2 真实值一致（单位没错，精度丢失在可接受范围——比如 V3 的 `decimal(12,2)` vs V2 的 `decimal(12,4)`，最后两位小数四舍五入是否影响财务对账？如果影响，V3 要把 DB 字段也改成 `Decimal(12,4)`，这是迁移时才会暴露的问题）
- 确认 V3 本地的旧 int 格式快照（迁移前存的 1299 = 12.99 元）和 V2 新格式的值（12.99 = 12.99 元）对比后完全一致，没有因为单位换算 bug 导致差 100 倍。

如果对账时发现差异率 < 0.01% 且差异都是"同步延迟导致 + 同步延迟时间在 24h 以内"，说明迁移成功。差异率 > 0.1% 立刻停止 Phase 3 的 DB 迁移，回滚到上一个版本，先排查解析 bug。
