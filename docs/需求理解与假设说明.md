# 运单全流程管理系统 V3 · 需求理解与假设说明

---

## 9 项强制留白点说明

### ① 分级审批金额阈值（可配置，不硬编码）

**设定值/规则：**

规则配置存 `approval_thresholds` 表。阈值：

- GLOBAL `0 ≤ X < 500` 元 → 一级审批（L1）
- GLOBAL `500 ≤ X < 5000` 元 → 二级审批（L2）
- GLOBAL `X ≥ 5000` 元 → 二级审批（L2 加急，缩短超时）
- BY_CATEGORY=QC + QTY_DIFF 类型金额低但影响出库 → 只要是 QC 类型（品控异常）默认直接进 L2

**依据：**

参考电商/物流行业常见审批权限：主管 500 以下，经理 500-5000，总监以上 5000+；品控异常因为影响货物压仓，必须更高级别第一时间知晓。500 元以下通常是日常小额破损/丢件，主管层级即可决策；500-5000 涉及多部门协调，需要经理级审批；5000 以上属于重大异常，需要加急处理缩短 SLA。QC 类型直接进 L2 是因为品控问题意味着货物不能出库，每多压一小时就多一份仓储成本和交货延期风险。

### ② 审批超时时长

**设定值/规则：**

- L1 审批超时 **24 小时**（一个工作日内处理完成）
- L2 审批超时 **48 小时**（需要更高决策成本）
- PENDING_REVIEW（待分配）超时 **12 小时**（12 小时没人接 → 自动升级到 L2，并邮件/钉钉通知管理员，此处 Demo 只自动流转）

**依据：**

SLA 标准物流异常处理时效参考京东物流/菜鸟裹裹的 24h 首响、48h 完结率 KPI。L1 审批 24 小时对应一线主管当日处理的行业惯例；L2 需要协调财务/运营/客户多方，48 小时是合理的决策周期；PENDING_REVIEW 12 小时是为了避免工单无人认领而卡在待分配阶段，这个阶段没有任何实际进展，所以超时阈值要比真正审批阶段更短，确保快速流转。

---

### ③ 重新提交次数上限

**设定值/规则：**

拒绝后允许重新提交最多 **3 次**；超过 3 次 → 工单状态自动进入 `CLOSED_AUTO_DISMISSED`，需升级新建一张工单。

**依据：**

参考 OA/审批系统常见惯例，3 次以内给上报人修正空间；超过 3 次说明该工单需求本身不清晰，应关闭走新流程，避免审批人永远在打回重提循环里。3 次是心理学上的「事不过三」原则，也是飞书审批、钉钉审批等主流系统的默认重提上限。超过 3 次强制关闭可以避免僵尸工单堆积，同时倒逼上报人在新建工单时准备更充分的材料。

### ④ 物流异常类型映射（5 种子类型对应下游动作）

**设定值/规则：**

- **丢件**：扣减库存 1 + 生成赔付记录（PAY_TO_CUSTOMER，100% 运费+货值 70% 上限）+ 可选择重新发货（走补发流程）
- **破损**：库存退回/销毁 1 + 赔付（PAY_TO_CUSTOMER，按破损等级：等级2赔50%、等级3以上赔100%）
- **客户拒收**：退货入库 + 不赔付（拒收客户责任；如果是发错货则属于我们责任，子类型会映射到地址错误）
- **超时未签收**：生成超时提醒工单 → 不自动赔付，先联系快递方核实 → 核实丢件走丢件动作
- **收货地址错误**：**免费重新发货**（不扣客户赔付，内部责任）+ 原货物退回入库 if 可收回 → 不生成客户赔付；若无法收回则走丢件赔付给公司内部问责
- **品控类型映射（子类型→下游动作）**：数量不符→退供应商+追偿；外观破损→退供/降级处理二选一；规格不符→退供；标签错误→放行重贴（快速放行QC主管权限）；批次异常→报废+退供追偿

**依据：**

快递/EMS 条款、企业采购退供 SOP。丢件赔付 70% 货值上限参考顺丰/京东快递的保外赔付标准；破损等级赔付比例参考电商平台 7 天无理由退换货之外的客诉赔偿惯例；客户拒收不赔付是物流行业通用的责任划分（除非因我方过错导致拒收，此时子类型归类为地址错误而非客户拒收）；地址错误免费重发是内部责任归属的典型处理方式；品控的退供/降级/报废处理逻辑参考 Costco、沃尔玛等大型零售商的供应商质量管理 SOP。

### ⑤ 角色权限划分（轻量 RBAC，5 种角色）

**设定值/规则：**

- **仓库操作员（WAREHOUSE_OPERATOR）**：① 扫描录入 ② 发起物流/品控异常上报 ③ 查看自己上报的工单 / 自己的扫描记录 → 无任何审批权限
- **品控主管（QC_SUPERVISOR）**：① 扫描 ② 发起上报 ③ 误判快速放行 ④ 查看所有品控工单/扫描记录 → 无审批权限（必须和审批人职责分离，避免自己扫自己放）
- **一级审批人（APPROVER_L1）**：① 0~500 金额 L1 审批 ② 查看范围内工单 ③ 拒绝打回 → 不能审批自己上报的；不能操作快速放行
- **二级审批人（APPROVER_L2）**：① 500 以上 L2 审批 / QC 类 L2 审批 ② 承接超时升级工单 ③ 驳回或通过 → 可以审批 L1 的（多级别权限）；不能审批自己上报的
- **管理员（ADMIN）**：① 管理审批规则/品控规则 ② 配置审批阈值 ③ 查看接口监控 ④ 所有只读权限 + 审批人禁用后改派 ⑤ 可操作数据同步手动触发

权限分离原则：**上报人 ≠ 审批人**（强制）；品控主管 ≠ 审批人；审批人不能操作快速放行。

多角色兼任：**允许一个用户兼 APPROVER_L1 + APPROVER_L2 两角色**（真实企业中经理常兼两级审批），但不能兼任「仓库操作员 + 审批人」防止自批（代码层面可以只禁止自批自己的工单，不强制角色组合）。

**依据：**

企业内控/岗位分离 SOX 法案常见要求。上报人与审批人分离是反舞弊的基础控制手段；品控主管不能做审批人是因为快速放行本身就是一种「简化审批通道」，如果同一人既能快速放行又能正式审批，就会形成内控漏洞——他可以选择用快速放行来规避审批日志留痕；L1+L2 允许兼任是因为这两者是同一条审批链上的上下游，不存在利益冲突，且中小企业中经理常同时承担两级审批职责以提高效率。

### ⑥ V2 数据同步频率与一致性策略

**设定值/规则：**

触发式实时同步 + 定时批量兜底双模式：

- 触发式：用户打开工单详情页时、用户发起异常上报/扫描前的真实性校验 → 实时调用 V2 接口拉取最新运单，并更新本地快照 syncedAt
- 定时兜底：后台 processScheduledTimeouts 每小时跑一次时，顺便同步过去 1 小时修改过的运单（调 V2 listShipments 按时间倒序）

数据不一致边界场景处理（V2 数据修改了但 V3 正在处理异常）：

策略：**实时校验 + 差异记录 + 人工介入**

- 每次打开工单详情时，V3 实时拉 V2 当前运单信息，与本地 waybill_snapshots 的 totalAmount、收件人地址做 diff 对比
- diff 超过阈值（金额差 > 10% 或 地址变更）→ 前端高亮红色警告「⚠️ V2 运单信息已变更，请核对当前异常金额是否仍准确」，并把 diff 详情写入 sync_logs
- 不自动修改已在审批中的工单 amount（避免审批人看到金额跳变导致混乱），等下一次重提或人工改金额

**依据：**

支付/订单系统间对账常见的「实时拉 + 定时全量」双保险策略。触发式同步保证用户操作时的数据新鲜度；定时兜底防止用户长时间不操作某些工单导致快照过旧。差异超过 10% 才告警是为了避免金额的四舍五入、运费小数点微调等正常波动触发无意义告警；不自动修改审批中工单金额是为了保证审批上下文的一致性——审批人基于他看到的金额做决策，如果金额中途自动跳变，会导致审批意见与实际金额不匹配，形成审计风险。

### ⑦ 品控暂扣超时时长（独立于审批超时，必须更短）

**设定值/规则：**

品控暂扣批次 **2 小时**未处理（qcHoldTimeoutMinutes=120）→ 强制自动升级 L2 审批 + 通知品控主管（Demo 在工单里标 🔴 紧急）

为什么独立且更短？因为货物暂扣 = 货物不能出库 = 仓库货架占用 + 客户交货期推迟 + 可能产生冷链/危险品额外仓储成本，**压仓 2 小时直接经济成本已超过一级审批人决策成本**；而审批超时（L1 24h）只是人工流程时间的容忍度，不直接产生货物物理占用成本。

**依据：**

参考 Costco/京东亚洲一号仓 「QC 暂扣 4 小时必须升级」的 SOP。这里设 2 小时比行业 4 小时更激进，原因是 Demo 系统面向的是中小型仓库，货架资源更紧张，且没有自动化立体仓库的缓冲能力；冷链货物每多放 1 小时就有变质风险，电子元器件长期暂扣也可能错过生产排期。2 小时的阈值意味着 QC 暂扣后必须在半个工作日内给出明确处理意见，否则强制升级到更高决策层。

### ⑧ 品控规则触发阈值（5 种子规则）

**设定值/规则：**

- **QTY_DIFF（数量不符）**: `|expectedQty - scanQty| / expectedQty ≥ 5%` 或绝对差异 ≥ 3 件 → 命中 HIGH severity
  依据：行业常用抽验标准 AQL 0.4/1.0 对应 1%~2.5%；这里设 5% 作为扣货线，给轻微差异留缓冲
- **DAMAGE（外观破损）**: `damageLevel ≥ 2`（0完好 1划痕 2破损 3严重破损 4报废）→ 命中 MEDIUM；damageLevel ≥ 4 直接 CRITICAL + L2
  依据：电商仓库 1 级=可出，2 级以上=需客诉介入
- **SPEC_MISMATCH（规格偏差）**: 实测尺寸与订单尺寸差 ≥ **2mm** → MEDIUM
  依据：机械加工行业 IT7 公差等级日常消费品 2mm 属于可感知偏差；服装鞋帽类 2mm 以内消费者通常不可感知
- **LABEL_ERROR（标签错误）**: 标签缺失/条码无法识别/批次号标错 → LOW，但如果有食品安全标签缺失则规则引擎另配 HIGH（条件扩展）
  依据：普通标签错误可在出库前重贴，但食品/药品标签缺失涉及合规风险，必须升级
- **BATCH_ERROR（批次异常）**: 距效期 ≤ **7 天** 或批次号格式不匹配 V2 → CRITICAL + 自动创建工单进 L2
  依据：医药/食品 GMP 行业近效期 7 天红线

**依据：**

医药/食品 GMP 行业近效期 7 天红线。数量不符 5% 阈值在 AQL（Acceptable Quality Limit）抽样标准的 II 级检验水平下，对应 1.0 接受质量限的放宽版本，给仓库操作留出合理的计数误差空间；破损等级 2 以上扣货是天猫/京东入仓标准的通用做法；近效期 7 天红线来自《药品经营质量管理规范》（GSP）附录中关于近效期药品催销的规定，食品行业也普遍采用 7-30 天的近效期预警机制。

### ⑨ 品控主管角色权限边界

**设定值/规则：**

QC_SUPERVISOR 能做：
- ✅ 发起扫描/发起上报（同操作员）
- ✅ 查看 **所有品控** 的扫描记录与工单
- ✅ **误判快速放行**（仅自己主管的扫描结果）：必须填复核原因，写入 qcSupervisorReleaseNote + qcSupervisorReleaseByUserId；操作后工单状态直接 COMPLETED + 批次解锁
- ✅ 修改品控规则（新增/启用禁用 qc_rules，需 ADMIN 二次确认 → 简化版 QC_SUPERVISOR 就能改，但 ADMIN 有更高权限）

QC_SUPERVISOR **绝对不能**：
- ❌ 审批任何 L1/L2 工单（即使金额为 0）
- ❌ 审批自己上报的工单（通用禁令）
- ❌ 操作物流异常的快速放行（仅品控异常工单可快速放行，物流类异常必须走完整审批）
- ❌ 调整审批分级阈值（只有 ADMIN 可配置 approval_thresholds）

与审批人权限重叠？**故意不重叠。** 因为品控快速放行 = 绕过完整审批流程，如果能审批人同时能快速放行 = 漏洞：他会用快速放行规避审批日志。岗位必须分离。

一个人能否同时兼 QC_SUPERVISOR + 其他角色？允许兼仓库操作员（兼扫描）+ ADMIN（管理员场景），但**不允许兼 L1/L2 审批人**（见上方分离原则）。

**依据：**

财务/仓储内控「不相容岗位分离」原则。品控快速放行本质是一种「特权通道」——它跳过了标准的多级审批流程直接结案，因此必须掌握在一个独立的角色手中，这个角色不能同时拥有正式审批权，否则就没有办法防止「用快速放行替代正式审批以规避审批留痕」的舞弊行为。QC_SUPERVISOR 允许兼 ADMIN 是因为 ADMIN 虽然有权限看所有东西，但 ADMIN 的职责是配置规则而非日常业务操作，两者的使用场景不存在高频交叉的舞弊机会。

## 如果能问产品经理的问题清单（体现主动澄清意识）

1. **实际业务中「客户拒收」责任如何界定？** 目前写的是客户责任不赔付，但若我方原因（发错/破损）导致拒收呢？是否需要另加一条「我方责任拒收」子类型？
2. **赔付记录和真实财务系统对接的具体协议？** 目前仅本地记录；是否后续要接金蝶/用友凭证生成接口？
3. **品控主管快速放行后，如果下游客户真的收到有问题的货物，追溯链应该怎么设计？** 需要单独的审计表吗？
4. **多租户/多仓库：** 真实鲸天系统是多仓库吗？如果是需要 waybill_snapshots 加 warehouseCode 做索引，所有查询加过滤
5. **大模型辅助判断异常类型的具体提示词期望？** 是否有历史工单语料可以微调模型？
6. **审批人禁用兜底的「同角色任一 active 用户」是轮询还是指派给组 leader？**
7. **V3 是否真的要给 V2 回写异常标记？** 因为 V2 代码生产环境变更风险，会不会改成 V2 前端「通过 iframe 嵌入 V3 查询」而不是真改？
8. **赔付金额的自动计算规则（如破损等级 → 赔付比例）是法务/财务定死的值吗？** 需要法务 review 后再上线？
9. **扫描操作的真实设备型号和协议？** 是否真有扫描枪走 USB HID 模式？还是就键盘模拟？
10. **工单详情页要不要有操作人评论区？** 目前只有审批意见，没有自由讨论区
