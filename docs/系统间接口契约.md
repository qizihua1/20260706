# 运单 V2 ↔ V3 系统间接口契约（v1.0）

---

## 1. 概述

运单 V2 系统（鲸天旧版）与运单 V3 系统（异常审批新版）为两个**独立部署、独立数据库**的服务。双方通过 **HTTP/JSON** 协议通信，所有接口强制 TLS（HTTPS）加密传输。

- V2 系统职责：运单生命周期管理、运单 CRUD、财务结算、客户信息等核心数据的权威源
- V3 系统职责：扫描品控、异常上报、分级审批、赔付/库存联动等异常流程管理
- 数据主权原则：运单基础信息（金额、地址、SKU 等）以 V2 为准；V3 只保存只读快照，发生冲突以 V2 实时拉取结果为准并告警人工介入

---

## 2. 鉴权机制

### 2.1 鉴权方式

所有跨系统接口通过 HTTP Header `x-api-key` 进行对称密钥鉴权。

### 2.2 密钥生成

```bash
# 生成 32 字节（256 位）随机十六进制密钥
openssl rand -hex 32
```

### 2.3 密钥分类

| 密钥变量名 | 所属方 | 用途 | 存放位置 |
|-----------|-------|-----|---------|
| `V3_EXTERNAL_API_KEY` | V2 侧生成，配置给 V3 | V3 → V2 接口调用时，V3 在 header 中携带，V2 校验 | V2 服务端环境变量 + V3 `.env.local` 的 `V2_API_KEY` |
| `V3_WEBHOOK_SECRET` | V3 侧生成，配置给 V2 | V2 → V3 Webhook 推送时，V2 在 header 中携带，V3 校验 | V3 服务端环境变量 + V2 Webhook 配置页 |

### 2.4 401 错误返回格式

```json
{
  "success": false,
  "error": {
    "code": "API_KEY_INVALID",
    "message": "x-api-key header is invalid or expired",
    "requestId": "req_abc123def456"
  }
}
```

`code` 枚举值：
- `API_KEY_MISSING`：Header 中未找到 `x-api-key`
- `API_KEY_INVALID`：密钥内容不匹配

## 3. 接口基础信息

### 3.1 Base URL

| 系统 | 环境 | Base URL |
|-----|-----|----------|
| V2 | Production | `https://code20200605.vercel.app` |
| V2 | Staging | `https://code20200605-staging.vercel.app` |
| V3 | Production | 用户部署域名，例如 `https://v3-approval.example.com` |
| V3 | Local | `http://localhost:3000` |

### 3.2 链路追踪

所有请求和响应必须在 Header 中携带 `x-request-id`：
- 格式：`req_` + 12 位随机十六进制（例如 `req_a1b2c3d4e5f6`）
- 生成方：调用方生成并写入 Header
- 回显要求：被调用方必须在 Response Header 中原样回显 `x-request-id`
- 日志要求：V3 侧所有 sync_logs 记录必须包含 `requestId` 字段，用 `grep "req_xxx"` 可还原完整跨系统调用链

### 3.3 超时与重试

- 单次请求超时：**10 秒**（`AbortController` 控制）
- 重试策略：最多 **2 次**指数退避重试，间隔 `500ms → 1000ms`
- 重试前提：仅对 `NETWORK_TIMEOUT`（无响应）和 `5xx` 错误重试；`401/400/422/200+exists=false` **不重试**

### 3.4 幂等性建议

- GET 请求天然幂等，无需额外处理
- PATCH `/exception-marker` 回写接口：建议 V2 侧按 `shipmentId + requestId` 做幂等去重，避免网络抖动导致 V3 重试时重复写入历史记录

### 4.2 GET /api/external/v1/shipments/{id} — 运单详情

**用途**：V3 打开工单详情实时拉取、上报前校验运单存在性。也支持通过 `byExternalCode=xxx` 查询（此时 `id` 填任意占位符，实际用 query 参数）。

**强调**：运单不存在时返回 **HTTP 200 + exists=false**，不是 404。这样 V3 可以区分「网络错误（真正的 404/5xx）」和「业务上运单不存在」，避免误报。

**Query 参数：**

| 参数 | 类型 | 必填 | 说明 |
|-----|------|-----|------|
| `byExternalCode` | string | 否 | 如果提供，则忽略 path 中的 id，按 externalCode 查询 |

**Request curl 示例：**

```bash
curl -X GET "https://code20200605.vercel.app/api/external/v1/shipments/shp_01h8xyz123" \
  -H "x-api-key: ${V2_API_KEY}" \
  -H "x-request-id: req_demo002detail01"

# 或按 externalCode 查询：
curl -X GET "https://code20200605.vercel.app/api/external/v1/shipments/placeholder?byExternalCode=YT202607060001" \
  -H "x-api-key: ${V2_API_KEY}" \
  -H "x-request-id: req_demo002detail02"
```

**Response JSON 示例（存在）：**

### 4.4 PATCH /api/external/v1/shipments/{id}/exception-marker — 异常状态回写 V2

**用途**：V3 审批通过/结案后，回写 V2 运单的异常标记字段，让 V2 前端也能看到此运单有异常记录（可选功能，Demo 中 V3 调用失败不阻塞主流程）。

**Request Body：**

```json

  "exceptionSubType": "LOST",

  "markedByUserId": "user_l2_001",
```

```bash
  -d '{"exceptionType":"LOGISTICS","exceptionSubType":"LOST","severity":"HIGH","ticketId":"tkt_abc123","ticketStatus":"COMPLETED","markedByUserId":"user_l2_001","remark":"丢件已赔付"}'
```

```json

    "shipmentId": "shp_01h8xyz123",
    "updatedAt": "2026-07-06T10:00:00.000Z"
```

## 5. V2 → V3 可选 Webhook（加分项）

### POST /api/webhook/v2-waybill-updates（V3 端监听）

**用途**：当 V2 运单金额/地址变更时主动推送 V3，避免 V3 每小时轮询才发现差异。

**触发条件（V2 侧判断）：**
- Shipment.totalAmount 变更
- Shipment.receiver* 地址字段任一变更
- Shipment.status 变更（便于 V3 同步快照状态）

**鉴权**：Header 中携带 `x-api-key` = `V3_WEBHOOK_SECRET`。

**Request Body 示例：**

```json
{
  "eventType": "SHIPMENT_UPDATED",
  "eventId": "evt_wh001abc",
  "shipmentId": "shp_01h8xyz123",
  "externalCode": "YT202607060001",
  "updatedFields": ["totalAmount", "receiverAddress"],
  "oldValues": {
    "totalAmount": 2580.00,
    "receiverAddress": "望京SOHO T1-2001"
  },
  "newValues": {
    "totalAmount": 3080.00,
    "receiverAddress": "望京SOHO T3-2501"
  },
  "occurredAt": "2026-07-06T09:30:00.000Z"
}
```

**V3 Response：**

```json
{ "success": true, "received": true, "requestId": "req_v3wh_001" }
```

**失败重试**：V2 侧收到非 2xx 则重试，最多 5 次指数退避（500ms/1s/2s/4s/8s），最后一次失败后写入死信队列人工介入。

## 6. 错误码规范表

| HTTP 状态 | 业务 code | 含义 | 典型触发场景 | V3 是否重试 |
|----------|-----------|------|------------|------------|
| 200 | — (exists=false) | 业务不存在（非错误） | 运单不存在但接口正常响应 | ❌ 不重试，按 NOT_FOUND 处理 |
| 400 | `BAD_PARAM` | 业务参数错误 | pageSize 超 200、日期格式非法 | ❌ 不重试，修复参数后再调 |
| 401 | `API_KEY_MISSING` | Header 缺失 x-api-key | 代码漏加 header | ❌ 不重试，修代码 |
| 401 | `API_KEY_INVALID` | 密钥值不匹配 | 环境变量配置错 | ❌ 不重试，修密钥 |
| 408 | `TIMEOUT` | V2 10s 超时 | V2 慢查询或网络抖动 | ✅ 指数退避重试 2 次 |
| 422 | `SHIPMENT_NOT_FOUND` | 回写时运单不存在 | PATCH exception-marker 传的 shipmentId V2 查不到 | ❌ 不重试，sync_logs 记录 |
| 500/502/503/504 | `V2_SERVER_ERROR` | V2 服务端错误 | V2 崩溃/网关异常/Vercel 冷启动超时 | ✅ 指数退避重试 2 次 |
| 任何 | `UNKNOWN` | 兜底分类 | 以上都不是 | 视情况而定，先记录日志 |

## 7. Request ID 链路追踪

V3 生成 → V2 接收 → V2 响应 header 返回 → sync_logs 全量记录。

**V3 侧 sync_logs 表关键字段：**

| 字段 | 示例 | 说明 |
|-----|------|-----|
| `requestId` | `req_a1b2c3d4e5f6` | 调用方生成的唯一 ID |
| `direction` | `V3_TO_V2` / `V2_TO_V3` | 调用方向 |
| `endpoint` | `GET /api/external/v1/shipments/xxx` | 目标接口 |
| `httpStatus` | `200` | HTTP 状态码 |
| `errorCategory` | `NETWORK_TIMEOUT` | 错误分类（见第 8 节） |
| `latencyMs` | `342` | 耗时毫秒 |
| `createdAt` | `2026-07-06T...` | 时间戳 |

用以下命令可还原跨系统调用链：

```bash
grep "req_a1b2c3d4e5f6" ./logs/sync-*.log
```

## 8. 错误日志分类（V3 侧 ErrorCategory）

V3 侧将所有 V2 接口调用结果归入 6 种 `ErrorCategory`，便于监控统计和告警聚合。

| 分类 | 枚举值 | 判定触发条件 |
|-----|-------|------------|
| 网络超时 | `NETWORK_TIMEOUT` | fetch `AbortError`（10s 超时触发）或 TCP 连接拒绝/无响应 |
| 鉴权失败 | `AUTH` | HTTP 401，子 code 为 API_KEY_MISSING / API_KEY_INVALID |
| 业务不存在 | `NOT_FOUND` | HTTP 200 且 `exists=false`（不是 404！） |
| 参数错误 | `BAD_PARAM` | HTTP 400，业务参数校验失败 |
| V2 服务端错误 | `V2_SERVER_ERROR` | HTTP 500/502/503/504 |
| 未知错误 | `UNKNOWN` | 以上分类均不匹配的兜底（例如 V2 返回了 HTML 而不是 JSON） |

## 9. 降级方案

当 V2 连续 3 次调用失败（或健康检查接口 `/api/external/v1/health` 失败）时，V3 自动进入降级模式：

1. UI 顶部显示橙色告警条：**「⚠️ V2 接口暂时不可用，当前运单信息为 {syncedAt} 时间的缓存，可能非最新」**
2. 首页列表、工单详情页等只读操作，展示本地 `waybill_snapshots` 表缓存的快照数据
3. 需要实时校验的操作（新建工单、扫描录入前校验 SKU 归属）：弹窗提示「V2 接口不可用，此操作需要联网校验，是否重试？」，提供「重试同步」按钮
4. 降级恢复条件：V3 后台每 2 分钟探测一次 V2 健康检查，连续 2 次成功后自动退出降级模式
5. 所有降级期间的操作均在 sync_logs 标 `degraded=true`，便于后续人工复核

## 10. 老系统二开意识（考点 5 强制）

### 10.1 接口版本策略

- URL 路径前缀 `/v1/`，后续升级 `/v2/`，老版本保留至少 **6 个月**并行
- 版本号在 URL 中显式携带，不放在 Header 里（更利于 CDN 缓存和运维排查）
- 例：未来 V2 出新版接口 → `GET /api/external/v2/shipments`，旧的 `/v1/` 继续运行至 V3 完全迁移

### 10.2 字段向后兼容原则

**只加新字段（都 optional），不删/不改老字段含义。**

- 例：V2 运单未来增加 `totalWeight`，新增字段 `totalWeight?: number` 即可，V3 没处理也不会炸
- 例：V2 要新增 `insuranceFee`，直接在 skuList 元素里加 `insuranceFee?: number`，老字段 `price`、`qty` 的含义和类型**绝不改变**
- V3 侧 JSON 解析用 `zod` schema 做宽松解析：未定义字段 `strip` 掉，optional 字段缺失不报错

### 10.3 字段类型变更兼容

比如 V2 `Shipment.status` 从 int（0/1/2）改成 enum 字符串（`CREATED/SHIPPED/DELIVERED`），或金额 `int→decimal`：

1. V2 侧先出 `/api/external/v2/shipments/` 新版接口
2. 新接口返回里**同时给 old 字段兼容**：`{ status: "SHIPPED", statusLegacy: 1, totalAmount: 2580.00, totalAmountCents: 258000 }`
3. 兼容期至少 **3 个月**，V2 侧监控 `/v1/` 调用量
4. 邮件/群聊通知 V3 方迁移，V3 改完并上线后，V2 监控到 `/v1/` 调用量连续 7 天为 0，再下线

### 10.4 灰度上线

- 先把 V2 `/external/v1` API 只给 **V3 staging 环境 API Key**；生产 Key 暂时 401 拒绝
- 观察 1 周 V3 staging 环境 sync_logs：要求无 4xx 异常（除了正常的 exists=false），5xx < 0.1%，P95 延迟 < 2s
- 满足后再开通生产 Key，先 1% 流量（V3 侧按 userId 或 requestId 哈希打标），再全量

### 10.5 破坏性变更 Checklist

任何破坏性变更（删字段、改字段类型、改字段含义、改默认值）必须过以下 Checklist：

1. ① **邮件通知对接人**：V3 技术负责人、V2 技术负责人、产品经理三方确认
2. ② **双版本并行 ≥ 1 个月**：新旧接口同时可调用，V3 有充足时间迁移
3. ③ **监控老版本调用量降为 0 后再下线**：V2 侧 Grafana 看板可见
4. ④ **保留回滚 commit**：V2 和 V3 仓库的破坏性变更 commit 独立可回滚，发布后 2 周内保留快速回滚能力
